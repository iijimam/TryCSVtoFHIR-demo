# CSV から FHIR リソースへの変換 --  設計の比較と考え方

## 結論

Python のみでも FHIR 変換は可能ですが、実運用では IRIS のような基盤を使うことで、開発・運用の負荷を大きく下げることができます。

| 観点 | Python のみ | IRIS |
|------|-----------|------|
| 責務 | アプリ側に集中 | サーバー側に分離 |
| 実装 | すべて自前 | [JSONTemplate](https://github.com/Intersystems-jp/JSONTemplate)で簡略化 |
| 学習コスト | 高い | 低い |
| 柔軟性 | 高い | 高い（設計次第） |
| 検証 | 基本のみ | 基本のみ<br>※リポジトリ作成時に加工することで指定プロファイルに対する検証を呼び出し可能 |
| 運用 | 自前構築 | プロダクションで管理 |
| 拡張性 | コード依存 | コンポーネント分離 |


## 設計の考え方

- 小規模・単発処理 → Python のみでも十分
- 継続運用・チーム開発 → IRIS の利用が有効

判断のポイントは「責務の配置」です。

- Python のみ：アプリケーション側にすべての責務が集中
- IRIS：データ処理・変換・連携をサーバー側に分離

この違いが、開発効率や保守性に大きく影響します。

### Python のみ

#### 利点

- 柔軟性が高い
- Python エコシステムが使える

理由：[fhir.resources](https://pypi.org/project/fhir.resources/) は、[pydantic](https://pydantic-docs.helpmanual.io/) V2 を基盤としているため、標準検証を含めた開発が行え、また、AI を利用したコード生成を取り入れることができる点です。

その分、処理の自由度は高く、用途に応じた柔軟な設計が可能です。
#### 考慮が必要な点

- FHIR構造の理解が必須
- 検証・ログ・運用を自前実装

理由：FHIR リソースを一から組み立てる必要がある場合、JSON 構造を熟知した上で実装を進める必要があります。

> コード例にもあるように、FHIR リソースの JSON 構造に合わせたディクショナリをテンプレート的に作成することで、JSON の深い階層を意識しない開発方法も検討できますが、使用する全 FHIR リソース分の用意が必要となります。

#### 向いているケース
- 小規模
- PoC

#### 補足
FHIR リソースを fhir.resources を利用して一から組み立てるのではなく、UI 表示用のデータ抽出や加工に利用する場合は、FHIR 構造に沿った直感的な操作ができるため有効です。

### IRIS

#### 利点
- [JSONTemplate](https://github.com/Intersystems-jp/JSONTemplate)で構造を意識せず書ける
- プロダクションで運用が可視化される
- サーバーサイドで処理完結可能

理由：
- 各種プロファイルに合わせて加工が必要な場合は、テンプレートクラスを継承することで、差分開発が行えます。
- プロダクションを利用することで、リポジトリへどのような要求が行われたかを、いつでもトレースできます。<br>また、処理の途中でエラーが発生した場合は、トレースを参照することで、エラー発生地点を素早く確認でき、原因となった元データがどのような情報であるか、すぐに解析することができ、復旧作業に必要となる情報収集から確認までの作業負担を軽減できます。

#### 考慮が必要な点

- 学習コスト（プロダクション）

    > メモ：開発言語は、ObjectScript と Python（Embedded Python）を選択できます。

#### 向いているケース

- 継続運用
- チーム開発
 
### まとめ

Python のみでも FHIR 変換は実装可能ですが、実装・運用・拡張の責務がすべてアプリケーション側に集中します。

IRIS を利用することで、これらの責務を基盤側に分離でき、開発効率・運用性・保守性を大きく向上させることができます。

IRIS を利用するためには、独自の開発方法を習得する必要はありますが、**最大の利点は、IRIS は、FHIR リポジトリとしても、FHIR サーバー機能としても、テーブルを格納するデータベースとしても利用できる点があげられます。**

本環境では、CSV ファイルの入力を例に FHIR リソースへの変換処理例をご紹介しましたが、データベースにあるテーブルのデータを利用した変換も同様に行えます。

変換したい対象データが、患者に紐づく過去の記録である場合、FHIR リソースを作成するために必要な情報は、かなり膨大なデータになる可能性があります。

もし、その元となるテーブルデータが IRIS に格納されている場合、同じサーバー上に Interoperability のプロダクションを追加することで、変換元の大量データを別の場所に転送することなく FHIR リソースへの変換が行え、目的の FHIR リポジトリへ REST 要求を行うことができます。

また、IRIS の FHIR リポジトリと同じ場所に、Interoperability のプロダクションを利用した FHIR 変換を用意する場合、変換からリポジトリ登録までの流れが、全て FHIR サーバ側で実行できます。

このように、データの所在と処理を同一基盤に集約できる点が、IRIS の大きな特徴です。
